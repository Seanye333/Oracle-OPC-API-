OAuth2 Authentication

The OPC API requires a Bearer token before you can call any endpoint. This lesson walks through every line of the auth code in OPCClient.

OAuth2 client_credentials requests-oauthlib Token refresh

Why client_credentials?

There are four OAuth2 grant types. client_credentials is used for server-to-server (machine-to-machine) communication with no human user involved. The pipeline runs as a background job — it never shows a browser login page.

requests

requests is the standard Python HTTP library. We import it here only for its Timeout and ConnectionError exception classes — used in the retry decorator on _get().

OAuth2Session

OAuth2Session is a wrapper around a regular requests.Session that automatically attaches the Authorization: Bearer <token> header to every request. You don't need to set it manually.

BackendApplicationClient

BackendApplicationClient configures the session to use the client_credentials grant — telling oauthlib "this is a machine, not a human user, no redirect URL needed".

tenacity

tenacity is a retry library. These four imports let you decorate a function with @retry(...) to automatically retry on specific exceptions with configurable backoff delays. We'll see this in Lesson 2.

The three auth methods

src/p6_client.py — lines 39–56
    def _create_session(self) -> OAuth2Session:
        client = BackendApplicationClient(client_id=self._cfg["client_id"])
        session = OAuth2Session(client=client)
        self._fetch_token(session)
        return session

    def _fetch_token(self, session: OAuth2Session) -> None:
        token = session.fetch_token(
            token_url=self._cfg["token_url"],
            client_id=self._cfg["client_id"],
            client_secret=self._cfg["client_secret"],
            scope=self._cfg["scope"] or None,
        )
        logger.info("OPC token acquired", extra={"expires_at": token.get("expires_at")})

    def _ensure_token(self) -> None:
        if self._session.token.get("expires_in", 0) < 60:
            self._fetch_token(self._session)
_create_session

Called once in __init__. Creates the OAuth2-aware session and immediately fetches the first token. The single underscore prefix means "private to this class".

BackendApplicationClient

This object configures the OAuth2 grant type. By passing only client_id (no redirect URI, no user scopes), it signals the client_credentials grant — machine-to-machine auth.

OAuth2Session(client=...)

Wraps requests.Session with the OAuth2 client. From now on, every session.get(url) call will include Authorization: Bearer <token> automatically — you never set the header yourself.

fetch token on creation

The token is fetched immediately when the session is created. This is called "eager" initialization — we want to fail fast at startup if credentials are wrong, not halfway through a pipeline run.

session.fetch_token

Makes a POST to the token_url with the credentials. Oracle IDCS validates them and returns a JSON response like {"access_token": "...", "expires_in": 900}. The session stores this token internally.

Structured logging

Notice extra={"expires_at": ...}. The JSON logger will include this as a top-level field in the log output — making it queryable in log aggregators like Splunk or CloudWatch.

_ensure_token — the refresh guard

expires_in is the seconds until expiry. If less than 60 seconds remain, we proactively refresh. This prevents the token from expiring during an API call mid-pipeline — a classic race condition bug.

🧠 Quick Check

Why does _ensure_token() refresh when expires_in < 60 rather than expires_in == 0?

To save network requests — refreshing early means fewer token calls overall.
To prevent the token from expiring mid-request — a 60-second buffer covers the time a slow API call might take.
The OPC API requires tokens to be refreshed every 60 seconds regardless of expiry.
The expires_in field is unreliable and rounds to the nearest minute.

HTTP Core & Retry

Every API call goes through _get(). It handles token refresh, URL construction, and automatic retry on network failures — without retrying on auth errors.

@retry decorator tenacity backoff raise_for_status()

The _get() method — annotated

src/p6_client.py — lines 62–73
    @retry(
        reraise=True,
        stop=stop_after_attempt(3),
        wait=wait_exponential(multiplier=2, min=2, max=30),
        retry=retry_if_exception_type((requests.Timeout, requests.ConnectionError)),
    )
    def _get(self, endpoint: str, params: dict | None = None) -> dict:
        self._ensure_token()
        url = f"{self._base_url}/{endpoint.lstrip('/')}"
        response = self._session.get(url, params=params, timeout=self._timeout)
        response.raise_for_status()
        return response.json()
@retry decorator — unpacked

reraise=True — after all retries fail, re-raise the original exception (not a tenacity wrapper).
stop_after_attempt(3) — try at most 3 times total.
wait_exponential(multiplier=2, min=2, max=30) — wait 2s after attempt 1, 4s after attempt 2, capped at 30s.
retry_if_exception_type(Timeout, ConnectionError) — ONLY retry on network errors. A 401 Unauthorized or 404 Not Found will NOT be retried — they're bugs, not transient failures.

_ensure_token() first

Called before every request — even on retries. If the token expired during the wait between retries, this will refresh it transparently before the next attempt.

URL construction

rstrip("/") on the base URL + lstrip("/") on the endpoint prevents double slashes. So both "projects" and "/projects" produce the same correct URL: https://tenant.../ppmrestapi/v2/projects.

session.get() — not requests.get()

We use self._session (an OAuth2Session), not the global requests.get(). The session automatically adds the Authorization: Bearer ... header and can reuse TCP connections (connection pooling).

raise_for_status()

Converts HTTP error codes into Python exceptions. 4xx → HTTPError (client error). 5xx → HTTPError (server error). A 200 OK does nothing. This ensures errors are never silently swallowed.

response.json()

Parses the response body as JSON and returns a Python dict or list. OPC returns either a plain list of records, or {"items": [...], "total": N} — the _paginate() method handles both shapes.

Why not retry on HTTP errors (4xx/5xx)?

A 401 means your credentials are wrong — retrying won't help. A 500 might be a bug in the server — retrying might make it worse. We only retry on transient network failures (timeout, dropped connection) where retrying actually makes sense.

Pagination Generator

The OPC API returns max 500 records per request. _paginate() is a Python generator that lazily fetches page after page until no records remain.

Python generator (yield) OPC filter syntax Lazy evaluation

How OPC pagination works

Fetching 1,200 projects with page_size=500
Request 1
500 records
offset=0
→
Request 2
500 records
offset=500
→
Request 3 (last)
200 records
offset=1000

The generator stops when a page returns fewer records than page_size (200 < 500).

_paginate() — annotated

src/p6_client.py — lines 75–96
    def _paginate(
        self, endpoint: str, fields: list[str], extra_filter: str | None = None
    ) -> Generator[list[dict], None, None]:
        """Yield pages of records from a paginated OPC endpoint."""
        offset = 0
        field_str = ",".join(fields)
        while True:
            params: dict = {
                "fields": field_str,
                "limit": self._page_size,
                "offset": offset,
            }
            if extra_filter:
                params["filter"] = extra_filter
            data = self._get(endpoint, params=params)
            records = data if isinstance(data, list) else data.get("items", [])
            if not records:
                break
            yield records
            if len(records) < self._page_size:
                break
            offset += self._page_size
Return type: Generator

Generator[list[dict], None, None] means: yields list[dict], the send() type is None, and the return value is None. This is the standard type hint for a generator that only yields, never receives values.

offset — page cursor

Starts at 0. After each full page of 500 records, it increments by 500. This is how the OPC API knows which "window" of data to return: page 1 = records 0–499, page 2 = 500–999, etc.

fields — comma-separated string

The OPC API requires fields as a single comma-separated string in the query param: ?fields=ObjectId,Name,Status. We join the list here once, outside the loop, to avoid repeated string work.

Query parameters

These become URL query params: ?fields=ObjectId,Name&limit=500&offset=0. The requests library URL-encodes them automatically when passed to session.get(url, params=params).

OPC filter syntax

OPC uses its own filter DSL: Status IN ("Active", "What-if") or ProjectObjectId = 1001. It's added as &filter=... to the URL. Not all endpoints support all filter operators — check the OPC API docs.

Response shape normalization

Some OPC endpoints return a plain list: [{...}, {...}]. Others return a wrapper object: {"items": [{...}], "total": 47}. This one-liner handles both shapes: if it's already a list, use it directly; otherwise extract "items".

Empty response = done

If the API returns an empty list (we've gone past the last record), break out of the loop immediately. This handles edge cases like a project with zero WBS nodes.

yield — the generator magic

yield records pauses the function and hands the current page to the caller. Execution resumes from here when the caller asks for the next value. The HTTP connection stays alive between yields. Nothing is buffered in memory beyond one page at a time.

Partial page = last page

If we got fewer records than page_size, this is the last page — no need to make another API call. Break immediately after yielding.

Advance the cursor

Only reached when the current page was full (500 records). Increment offset by page_size to request the next window of records.

_get_all() — collects everything

src/p6_client.py — lines 98–104
    def _get_all(self, endpoint: str, fields: list[str],
                 extra_filter: str | None = None) -> list[dict]:
        """Collect all paginated records into a single list."""
        result: list[dict] = []
        for page in self._paginate(endpoint, fields, extra_filter):
            result.extend(page)
        logger.info("Fetched records", extra={"endpoint": endpoint, "count": len(result)})
        return result
Generator consumption

The for page in self._paginate(...) loop drives the generator. Each iteration triggers the next API call. result.extend(page) appends each page's records to the growing list. After the loop, result contains every record from every page.

Generator vs. list — why does it matter?
  • A list-based approach would fetch ALL pages first, then return — storing everything in memory at once.
  • A generator yields one page at a time. If a downstream task can process page 1 while page 2 is fetching, you get true streaming. It also means you can break early and never fetch remaining pages.
  • For 50,000 activities, this is the difference between loading ~40MB into RAM all at once vs. processing 500 records at a time.

def _paginate(self, endpoint, fields, extra_filter=None):
    offset = 0                     # start from the beginning
    field_str = ",".join(fields)   # OPC expects comma-separated string
    while True:
        params = {"fields": field_str, "limit": self._page_size,
                  "offset": offset}

            break                      # partial page = last page
        offset += self._page_size   # advance cursor

    def get_projects(self, status_filter: list[str] | None = None) -> list[dict]:
        fields = self._cfg["fields"]["project_fields"]
        f = None
        if status_filter:
            statuses = ",".join(f'"{s}"' for s in status_filter)
            f = f"Status IN ({statuses})"

    def get_wbs(self, project_object_id: int) -> list[dict]:
        fields = self._cfg["fields"]["wbs_fields"]
        return self._get_all("wbs", fields,
            extra_filter=f"ProjectObjectId = {project_object_id}")

    def get_activities(self, project_object_id: int) -> list[dict]:
        fields = self._cfg["fields"]["activity_fields"]
        return self._get_all("activities", fields,
            extra_filter=f"ProjectObjectId = {project_object_id}")

    def get_resource_assignments(self, project_object_id: int) -> list[dict]:
        fields = self._cfg["fields"]["resource_assignment_fields"]
        return self._get_all("activityresourceassignments", fields,
            extra_filter=f"ProjectObjectId = {project_object_id}")

    def get_resources(self) -> list[dict]:
        fields = self._cfg["fields"]["resource_fields"]
        return self._get_all("resources", fields)

    def get_roles(self) -> list[dict]:
        role_fields = ["ObjectId", "Id", "Name", "PricePerUnit",
                       "UnitOfMeasure", "IsActive"]
        return self._get_all("roles", role_fields)
get_projects — optional filter

The status_filter=None default means "fetch all projects". When called with ["Active", "What-if"], it builds a filter string. The fields list comes from config.yaml so it can be changed without touching code.

Building the OPC filter string

OPC expects: Status IN ("Active","What-if"). The generator expression f'"{s}"' wraps each status in double quotes, then ",".join(...) produces "Active","What-if". Finally the f-string wraps it in IN (...).

get_wbs — per-project scope

WBS nodes belong to exactly one project. The filter ProjectObjectId = 1001 limits the OPC query server-side. This is much more efficient than fetching all WBS nodes and filtering in Python.

get_resources — no filter

Resources (employees, equipment) are tenant-wide master data — not scoped per project. A resource can be assigned to multiple projects, so we fetch the complete list once.

get_roles — hardcoded fields

Role fields are hardcoded here instead of coming from config because roles have a stable, small schema that doesn't need configuration flexibility. This is a valid tradeoff for simpler code.

Context Manager Pattern

        self._session.close()

        return self

    def __exit__(self, *_: Any) -> None:

# This enables the "with" statement:
with OPCClient(config) as client:
    projects = client.get_projects()
# session.close() called automatically here, even if an exception occurred
close()

Closes the underlying OAuth2Session (which inherits from requests.Session). This releases the TCP connection pool, cleaning up file descriptors and network resources. Important in long-running Airflow workers.

__enter__

Returns self — the OPCClient instance. This is what's assigned to the as client variable in with OPCClient() as client:. Called when the with block begins.

__exit__

Called when the with block exits — whether normally or via exception. The *_ absorbs the exception type/value/traceback arguments (we don't need them since we always just close). Returning None (implicitly) means exceptions are re-raised.

Always use the context manager

Use with OPCClient() as client: instead of creating a client and calling close() manually. The with statement guarantees cleanup even if an exception occurs inside the block — you can't accidentally leak a connection.

The Extractors

Three extractor classes sit between the raw OPCClient and the rest of the pipeline. They orchestrate multi-entity fetches, log progress, and return clean data dictionaries.

ProjectExtractor CostExtractor ResourceExtractor

Why a separate extractor layer?

Single Responsibility Principle

OPCClient knows about HTTP, OAuth2, and pagination. Extractors know about business logic: which projects to include, how to loop per-project, and what to log. Separating these makes each class simpler to test, modify, and understand.

ProjectExtractor — annotated

src/extractors/project_extractor.py
class ProjectExtractor:
    def __init__(self, client: OPCClient, status_filter: list[str] | None = None):
        self._client = client
        self._status_filter = status_filter or ["Active", "What-if"]

    def extract_projects(self) -> list[dict]:
        logger.info("Extracting projects", extra={"status_filter": self._status_filter})
        projects = self._client.get_projects(self._status_filter)
        logger.info("Projects extracted", extra={"count": len(projects)})
        return projects

    def extract_wbs(self, project_ids: list[int]) -> list[dict]:
        all_wbs: list[dict] = []
        for pid in project_ids:
            wbs = self._client.get_wbs(pid)
            all_wbs.extend(wbs)
            logger.debug("WBS fetched", extra={"project_id": pid, "wbs_count": len(wbs)})
        logger.info("WBS extraction complete", extra={"total": len(all_wbs)})
        return all_wbs

    def extract_all(self) -> dict[str, list[dict]]:
        projects  = self.extract_projects()
        project_ids = [p["ObjectId"] for p in projects]
        wbs       = self.extract_wbs(project_ids)
        activities = self.extract_activities(project_ids)
        return {
            "projects": projects,
            "wbs": wbs,
            "activities": activities,
        }
Constructor — dependency injection

OPCClient is passed in, not created internally. This is dependency injection: the caller controls the client (with its credentials/config), and the extractor just uses it. This makes it trivial to pass a mock client in tests without any patching.

extract_projects — book-ended logging

One logger.info before and one after the actual call. The first tells you "we're starting, with these parameters". The second tells you "we finished, got N records". This pattern makes pipeline logs easy to debug when something hangs or returns 0 records.

extract_wbs — per-project loop

WBS nodes are fetched one project at a time (the OPC API requires a project filter). all_wbs.extend(wbs) flattens the per-project lists into one combined list. The logger.debug inside the loop gives per-project counts for fine-grained tracing.

extract_all — orchestrator method

Convenience method that runs the full sequence. Notice the dependency chain: projects must be fetched first to get the project_ids, which are then used to fetch WBS and activities. This is explicit ordering — not concurrent.

class CostExtractor:
    def extract_project_cost(self, projects: list[dict]) -> list[dict]:
        cost_fields = [
            "ObjectId", "Id", "Name", "DataDate",
            "TotalBudgetedCost", "ActualTotalCost",
            "EarnedValueCost", "PlannedValueCost",
            "CostPerformanceIndex", "SchedulePerformanceIndex",
            "EstimateAtCompletion", "BudgetAtCompletion", "SummaryBudget",
        ]
        records = []
        for p in projects:
            record = {k: p.get(k) for k in cost_fields}
            records.append(record)
        return records

    def extract_activity_cost(self, activities: list[dict]) -> list[dict]:
        cost_fields = [
            "ObjectId", "ProjectObjectId", "WBSObjectId", "Id", "Name",
            "PlannedTotalCost", "ActualTotalCost", "BudgetAtCompletion",
            "EarnedValueCost", "PlannedDuration", "RemainingDuration",
            "PercentComplete", "PlannedStartDate", "PlannedFinishDate",
            "ActualStartDate", "ActualFinishDate", "Status",
        ]
        return [{k: a.get(k) for k in cost_fields} for a in activities]
Module-level constant

_EV_SPREAD_FIELDS is a module-level constant (underscore = "private to this module"). It lists EV spread fields available in OPC for period-level data — distinct from the project-level fields extracted in extract_project_cost. These could be used for future period-level EV time series extraction.

Dict comprehension with .get()

{k: p.get(k) for k in cost_fields} creates a new dict with only the keys we care about. Using .get(k) instead of p[k] is critical — if OPC doesn't return a field for a particular project (e.g., no budget set), .get() returns None instead of raising KeyError.

List comprehension shorthand

Exactly the same logic as the for-loop above, but written as a one-liner list comprehension. Both are equivalent — use whichever is more readable for the team. The list comprehension is slightly faster in CPython due to bytecode optimization.

ResourceExtractor

src/extractors/resource_extractor.py
class ResourceExtractor:
    def __init__(self, client: OPCClient):
        self._client = client

    def extract_resources(self) -> list[dict]:
        resources = self._client.get_resources()   # Tenant-wide
        logger.info("Resources extracted", extra={"count": len(resources)})
        return resources

    def extract_roles(self) -> list[dict]:
        roles = self._client.get_roles()
        logger.info("Roles extracted", extra={"count": len(roles)})
        return roles

    def extract_assignments(self, project_ids: list[int]) -> list[dict]:
        all_assignments: list[dict] = []
        for pid in project_ids:
            assignments = self._client.get_resource_assignments(pid)
            all_assignments.extend(assignments)
            logger.debug("Assignments fetched",
                extra={"project_id": pid, "count": len(assignments)})
        logger.info("All assignments extracted", extra={"total": len(all_assignments)})
        return all_assignments

    def extract_all(self, project_ids: list[int]) -> dict:
        return {
            "resources":            self.extract_resources(),
            "roles":                self.extract_roles(),
            "resource_assignments": self.extract_assignments(project_ids),
        }
Tenant-wide vs. per-project

Resources and roles are master data — they exist at the tenant level, not scoped to any project. One employee can work on dozens of projects. So we fetch them once for the whole tenant and cross-reference by ResourceObjectId in the assignments.

extract_assignments — same loop pattern

Identical loop structure to ProjectExtractor.extract_wbs(). This is intentional — a consistent pattern across extractors makes the code predictable. A developer who knows one extractor can immediately read any other.

Putting it all together

Complete API extraction flow (as used in the Airflow DAG)
from src.p6_client import OPCClient
from src.utils.config import get_opc_config
from src.extractors.project_extractor import ProjectExtractor

config = get_opc_config()

with OPCClient(config) as client:
    # ── Step 1: Extract project hierarchy ──────────────────
    proj_ext = ProjectExtractor(client, status_filter=["Active", "What-if"])
    data = proj_ext.extract_all()
    # data = {"projects": [...], "wbs": [...], "activities": [...]}

    project_ids = [p["ObjectId"] for p in data["projects"]]
    print(f"Projects: {len(data['projects'])}")
    print(f"WBS nodes: {len(data['wbs'])}")

🧠 Final Quiz

In CostExtractor.extract_project_cost(), why does it use p.get(k) instead of p[k]?

.get() is faster than bracket indexing for large dictionaries.
.get(k) returns None if the key is missing, preventing a KeyError when OPC doesn't return a field for a particular project.
OPC field names can contain special characters that bracket syntax can't handle.
It allows the extractor to work with both dicts and objects.
